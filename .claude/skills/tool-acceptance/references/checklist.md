# Tool Acceptance Checklist

按这个清单执行单接口验收。

## 1. 基本信息

- 接口名
- 验收日期
- 验收目标
- 验收范围
- 环境前提
  - 本地服务是否启动
  - 认证信息是否可用
  - 代理是否影响上游访问

## 2. 全链路代码阅读

至少覆盖这些层：

- ToolSpec
- service
- provider
- CLI
- HTTP API
- MCP
- Web 前端/管理后台
- health / consistency
- 相关测试

输出应至少回答：

- 这个接口的参数和返回字段是什么
- 默认 provider 链是什么
- 对外暴露路径有哪些
- 当前有哪些测试覆盖

## 3. 验收方案和计划

在真正执行前，先给出：

- 验收目标
- 验收范围
- 分阶段检查项
- 每阶段使用什么方法验证
- 通过标准

## 4. 上游官方文档与原始接口验证

这是 provider 验收前置步骤。**禁止跳过，禁止用代码推断替代真实调用。**

对每个 provider：

1. 找到官方文档或真相源入口
2. 定位当前接口对应的具体文档或源码定义
3. **真实调用原始接口，`print(df.columns.tolist())` 记录实际字段名**
4. **打印至少 1 行样本数据，记录实际值和量级**
5. 记录原始字段、单位、时间语义、错误行为

必须明确：

- 用的是哪个上游接口
- 字段语义是什么
- 单位是什么（文档描述 vs 实际值量级，两者不一致时以实际值为准）
- 是否包含未完成周期
- 是否需要认证

如果没有稳定官方文档：

- 记录实际 endpoint
- 记录真实 HTTP 响应
- 把项目内 provider 实现视为”适配层”，不是最终真相源

## 5. Provider 验收

逐个 provider 验：

- 原始接口是否可调用
- **`r.get(“xxx”)` 中的每个字段名是否与第 4 步的 `df.columns` 完全一致**
- 入参映射是否正确
- 输出字段是否正确
- **单位转换是否正确（对照第 4 步样本值验证量级）**
- 时间字段是否正确
- 错误分类是否正确

输出：

- 每个 provider 的结论：通过 / 部分通过 / 不通过 / 未执行
- 关键证据
- 根因判断

## 6. Service 验收

检查：

- provider 链是否符合设计
- fallback 是否符合设计
- 标准化结果是否一致
- health / consistency 是否符合预期

### 跨源一致性门禁（多源接口必须执行）

多源接口必须在 service 验收阶段完成以下检查：

1. **字段集对齐**：逐个对比所有 provider 返回的字段集。如果某字段只有一个源有：
   - 另一个源能补 → 补上映射
   - 另一个源确实没有 → 从所有源删除该字段
   - **不允许带着字段缺失差异签收**
2. **执行 health probe**：用 `finance-data health <tool>` 或 dashboard”探测此接口”，确认 consistency = “一致”（0 差异）
3. **修复后必须重新探测**：改了 provider 代码后，必须重跑 probe 验证结果，不能只看代码

如果是单源 service：

- 明确写出”当前单源，无跨源一致性要求”

## 7. CLI 验收

至少覆盖：

- describe
- 默认参数调用
- 显式参数调用
- provider 直连调用（如果对外支持）

检查：

- 参数契约是否正确
- 默认值是否生效
- 返回结构是否正确
- 错误是否正确暴露

## 8. HTTP API 验收

至少覆盖：

- 默认参数调用
- 显式参数调用
- provider 指定调用（如果后台支持）

检查：

- 请求体和响应体是否正确
- 默认参数是否补齐
- 错误响应是否正确

## 9. MCP 验收

至少覆盖：

- 默认参数调用
- 显式参数调用

检查：

- tool 签名是否正确
- 返回 JSON 结构是否正确
- 默认值和动态默认值是否生效

## 10. Web 前端 Playwright 验收

必须真实访问页面。

至少覆盖：

- 打开页面
- 选择目标接口
- 检查默认值展示
- 填写参数
- 提交调用
- 校验成功渲染

必要时补：

- provider 对比模式
- 下拉枚举选择
- 错误提示渲染

检查重点：

- 页面字段是否完整渲染
- 数据源卡片是否正确
- 表格或卡片的关键字段是否正确
- 页面是否真实反映后端错误

## 11. 测试与回归

至少记录：

- 跑了哪些测试
- 是否通过
- 有无遗漏

如果没有执行测试：

- 明确写原因

## 12. 最终结论

### 签收前置条件（全部满足才能签收）

1. **语义传播判定**
   - provider 层是否有语义问题（单位、字段含义、数据映射错误）？→ 如果有，service 及所有交付层自动"不通过"
   - service 层是否有语义问题（跨源不一致未处理、fallback 语义漂移）？→ 如果有，所有交付层自动"不通过"
   - **"能调通" ≠ "通过"**：返回 HTTP 200 或页面渲染正常，但数据语义有错 → 不通过

2. **契约一致性**
   - ToolSpec metadata（source_priority 等）是否匹配实际 service
   - MCP docstring 数据源描述是否匹配实际 provider 链
   - CLI describe 输出是否匹配实际参数和 provider

3. **跨源一致性（多源接口必须满足）**
   - 所有 provider 返回相同字段集（0 缺失）
   - health probe consistency = "一致"（0 差异）
   - 以上必须是修复后重新探测的结果，不能用修复前的缓存数据

### 结论格式

必须给出：

- provider：通过 / 不通过
- service：通过 / 不通过
- CLI：通过 / 不通过
- HTTP API：通过 / 不通过
- MCP：通过 / 不通过
- Web 前端：通过 / 不通过 / 未执行
- 最终状态：已签收 / 未签收

如果某层因底层语义传播而不通过，必须标注：
- `不通过（语义传播：provider 层 XXX 问题）`

最终报告结构见 [report-template.md](report-template.md)。
